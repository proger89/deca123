# SafeSort — цифровая сортировочная ячейка для задачи 3 RoboZone/Ozon

## Запуск демо — Docker в приоритете

Требуется запущенный Docker Desktop. В Windows достаточно дважды нажать `start-demo.cmd` или выполнить одну команду из корня репозитория:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\start-demo.ps1
```

Скрипт собирает и запускает контейнер, ждёт успешной проверки готовности и только затем автоматически открывает `http://127.0.0.1:4173/` в браузере. STL анализируется локально в браузере и не отправляется на сервер.

Для CI или сервера, где браузер открывать не нужно:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\start-demo.ps1 -NoBrowser
```

Прямой серверный запуск и остановка:

```powershell
docker compose up --build --detach demo
docker compose down
```

Порт можно изменить: `powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\start-demo.ps1 -Port 8080`. Запуск без Docker оставлен только как резервный вариант в [инструкции браузерного демо](web/README.md).

Режим GPU по умолчанию — `Auto`: скрипт проверяет NVIDIA именно внутри Docker, передаёт доступную видеокарту контейнеру и без ошибки откатывается на CPU, если GPU или NVIDIA Container Runtime отсутствуют. Явный выбор: `-GpuMode Gpu` или `-GpuMode Cpu`. Браузерный 3D-просмотр использует ускорение самого Chrome; для физического Webots-прогона с тем же автоматическим выбором выполните:

```powershell
.\scripts\run-webots.ps1 -GpuMode Auto
```

Передача GPU и фактически выбранный Webots-рендерер — разные факты. Они записываются честно: контейнер может видеть NVIDIA, но headless Webots под Xvfb всё равно выбрать Mesa/CPU. Такой запуск остаётся корректным и воспроизводимым, просто медленнее.

SafeSort определяет категорию неизвестного товара по виртуальным датчикам глубины, а затем физически направляет его в зону B, C или D на конвейере Webots. Исполняющий алгоритм не получает имя STL, имя объекта сцены или эталонный класс. Успех фиксируется только после сигнала датчика нужного выхода.

> Главная демонстрация проекта — виртуальный конвейер в Webots. Браузерная страница нужна жюри для быстрой загрузки собственного STL и наглядного объяснения решения; она не заменяет физическую симуляцию.

## Что именно решает система

### Что означают B, C и D

| Зона | Когда товар направляется туда | Коротко в интерфейсе |
|---|---|---|
| **B** | Все три габарита строго внутри допустимых границ и `K ≤ 0,8` | стандартный поток |
| **C** | Хотя бы один габарит не проходит строгую проверку; габариты проверяются раньше формы | габаритный товар |
| **D** | Габариты допустимы, но поперечное сечение круглое: `K > 0,8` | круглая форма |

Пороги и порядок B/C/D заданы организаторами. Это правильный ответ, который обязана получить любая команда, а не уникальная часть проекта.

### В чём наша задумка и отличие

SafeSort не хранит таблицу заранее известных моделей и не угадывает класс по имени файла. Уникальность решения — в полном проверяемом цикле для неизвестного товара:

1. пять синхронных виртуальных датчиков глубины измеряют товар с разных сторон;
2. алгоритм восстанавливает габариты и коэффициент формы из измерений, устойчиво к повороту STL;
3. отдельный модуль применяет официальное правило B/C/D и честно обрабатывает пограничную неопределённость;
4. две пассивно-безопасные заслонки физически направляют товар по виртуальному конвейеру;
5. успех разрешён только после сигнала фактического датчика нужного выхода;
6. при сбое система не выбирает опасный путь B, а переходит в безопасный маршрут или защёлкнутый `FAULT`;
7. жюри может загрузить собственный переименованный STL, увидеть измерения, маршрут, анимацию, журнал и скачать JSON-отчёт.

Иными словами: **правило классификации дано в задании; наша инженерная ценность — как независимо измерить неизвестный товар, исполнить решение на конвейере, пережить отказ и доказать результат**.

Официальные правила применяются строго и в заданном порядке:

1. Три габарита товара должны быть больше 10 мм и меньше 450×320×320 мм. Равенство границе не проходит: такой товар относится к C.
2. Только после проверки габаритов вычисляется коэффициент круглости `K = r_вписанной / R_описанной`.
3. При `K > 0,8` товар относится к D; при `K ≤ 0,8` — к B.
4. Если измерение около границы ненадёжно, система не угадывает B: неопределённость по габаритам безопасно направляется в C, по форме — в D.

Полная цепочка:

```text
STL товара → конвейер A → 5 синхронных датчиков глубины
→ облако точек → габариты и K → правило B/C/D
→ две пассивно-безопасные заслонки → датчик выхода → SUCCESS/FAULT
```

## Два уровня демонстрации

| Уровень | Что видит жюри | Для чего нужен |
|---|---|---|
| Браузерный демонстратор | Локальная загрузка STL, 3D-просмотр, измерения, решение B/C/D и анимация маршрута | Быстро проверить собственный или переименованный файл без установки ПО |
| Webots | Полноразмерная виртуальная площадка, движущийся товар, пять RangeFinder, две заслонки, зоны B/C/D, отказы и датчики выхода | Авторитетное доказательство физического цикла и отказобезопасности |

STL — треугольная поверхность отдельного товара. STEP — редактируемая CAD-геометрия товара; это не «файл конвейера». Конвейер, датчики, заслонки и зоны описаны в мире Webots и `FailSafeGate.proto`.

## Быстрый старт для жюри

Требования: Git, Python 3.12 и запущенный Docker Desktop. Локальная установка Webots и GPU не требуются. Первый образ может скачиваться дольше; это время считается отдельно.

```powershell
git clone https://github.com/proger89/deca123.git
cd deca123
python run_scenario.py suite --profile release --repeat 3 --record-video --output artifacts/release
```

Команда собирает закреплённый CPU-образ, запускает его без сети и создаёт проверяемый набор: журналы, CSV, HTML/PDF-отчёт, панель, MP4, манифест и контрольные суммы. Любая красная проверка, отсутствие обязательного файла или подтверждения выхода даёт ненулевой код завершения.

Отдельный физический smoke-тест полного цикла:

```powershell
python run_scenario.py run --scenario scenarios/smoke/unknown_stl_b.yaml --seed 42 --output artifacts/smoke
python run_scenario.py verify --bundle artifacts/smoke
```

Полная финальная проверка из чистых клонов:

```powershell
python run_scenario.py release check --repeat 3 --clean-clone
python run_scenario.py quality --checks all
python tools/check_submission.py --bundle submission/final
```

## Текущий уровень доказательств

Проект не смешивает Webots-физику с численными проверками. Численные результаты ниже пересчитаны из первичных CSV/JSONL. Финальные физические трассы Webots принимаются только после повторного запуска с актуальной калибровкой, совпадения `requested route` и `confirmed exit` и прохождения missing-exit canary. Старые трассы до исправления калибровки в итоговые заявления не входят.

| Проверка | Тип доказательства | Что находится в текущем артефакте |
|---|---|---|
| Неизвестный товар A → B/C/D → датчик выхода | Webots physical | финальный статус пока не заявлен: нужны новые post-calibration B/C/D и missing-exit traces |
| 10 564 маршрута, accuracy/coverage 0,9985806975 | пятиракурсный sensor model + continuous-time conveyor proxy | 14 консервативных abstain из 9864 классифицируемых; это не реальные STL и не Webots-маршруты |
| Смешанный proxy-поток | тот же численный harness | 3486/3500 = 0,996; unsafe-to-B/lost/duplicates = 0 внутри модели |
| p95 габаритов 0,578523 мм; p95 K 0,00813178 | synthetic geometry внутри sensor model | модельные ошибки; реальная STL/RangeFinder проверка ведётся отдельно |
| 5143 входа/выхода за модельный час | аналитический и дискретно-событийный hour-flow | профиль 1 м/с, шаг 700 мм, конечная очередь 0, максимум ожидания 2; не физический час Webots |
| 200 потерь питания, возврат ≤0,342 с | численная physics-proxy модель заслонки | решение уравнения движения и сравнение с расчётом; Webots smoke проверяет отдельный штатный цикл |
| Три одинаковых SHA-256 | детерминизм замороженной release-сводки | равенство семантических сводок, а не независимое повторное измерение всех ранних метрик |

Профиль 7200 товаров/ч имеет статус `UNSUPPORTED`: шаг 500 мм не прошёл полную квалификацию. Профиль 5143 товара/ч остаётся расчётным/discrete-event результатом, пока не подтверждён отдельным длительным Webots-прогоном.

## Архитектура безопасности

- `RuntimeController` читает только виртуальные датчики, калибровку и энкодер; Supervisor, исходная модель и oracle запрещены архитектурной проверкой.
- `EvaluatorSupervisor` получает события односторонне и добавляет эталон только после действия runtime.
- Решения привязаны к монотонному `item_seq` и позиции энкодера, а не к задержке `sleep` или имени файла.
- Первая заслонка без питания механически возвращается к C, вторая — к D. Для B нужны два подтверждённых датчика положения.
- `SUCCESS` невозможен без совпавшего датчика выхода; потеря выхода, заклинивание или несоответствие маршрута дают защёлкнутый `FAULT`.
- Файлы и метрики набора связаны SHA-256; намеренная порча отчёта обнаруживается проверяющей командой.

## Ограничения и честные границы

- Это цифровая симуляция, а не испытание собранного промышленного оборудования.
- Браузер анализирует геометрию STL непосредственно на устройстве пользователя; он не доказывает работу пяти RangeFinder. Для такого заявления нужен отдельный зелёный post-calibration Webots-прогон.
- Сырые PDF, чат, STL и STEP организатора не публикуются: в комплекте нет явной лицензии на распространение. Хэши источников сохранены.
- GPU не требуется и не входит в базовые заявления.
- Любое итоговое число принимается только вместе с манифестом, seed, первичным журналом и контрольной суммой.

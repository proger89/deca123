# Архитектура SafeSort

## Назначение

SafeSort — цифровая модель сортировочного узла: неизвестный товар движется по конвейеру от зоны A, измеряется пятью виртуальными датчиками глубины и физически направляется в B, C или D. Архитектура специально разделяет управление и оценивание, чтобы алгоритм не мог узнать правильный ответ из STL или сцены.

## Контур выполнения

```text
                         только события, одно направление
┌──────────────────┐  Emitter ─────────────────────────────► ┌───────────────────┐
│ RuntimeController│                                        │EvaluatorSupervisor│
│                  │                                        │                   │
│ 5 RangeFinder    │                                        │точная истина      │
│ энкодер и вход   │                                        │после действия     │
│ геометрия        │                                        │выходы и метрики   │
│ RuleEngine       │                                        │fault injection    │
│ ItemLedger       │                                        └───────────────────┘
│ две заслонки     │
│ датчики выходов  │
└──────────────────┘
          │
          ▼
 A → измерение → заслонка габаритов (покой к C)
                → заслонка формы (покой к D) → B
```

`RuntimeController` и `EvaluatorSupervisor` — отдельные контроллеры Webots. Общим остаётся только пакет контрактов событий. Обратного канала от evaluator к runtime нет.

## Поток одного товара

1. Входной фотоэлектрический датчик создаёт новый монотонный `item_seq`.
2. Энкодер задаёт позицию товара и крайний момент принятия решения.
3. Пять RangeFinder формируют `FrameBundle` одного simulation tick и одной позиции энкодера.
4. Из калиброванных точек строится объединённое облако, оцениваются ориентированный ограничивающий параллелепипед и круглые поперечные сечения.
5. Консервативный слой учитывает интервалы ±5 мм и ±0,03 K. Пересечение порога не разрешает B.
6. Точный `RuleEngine` сначала проверяет габариты, затем K.
7. Решение навсегда связывается с `item_seq`; более поздний результат другого worker не может его изменить.
8. Первая заслонка отделяет C, вторая — D. Для прохода в B нужны два разрешения и два подтверждения положения.
9. Только совпавший датчик выхода переводит товар в `EXIT_CONFIRMED/SUCCESS`. Отсутствующий или неверный выход даёт `FAULT` и блокирует следующий выпуск.

## Компоненты

| Компонент | Ответственность | Не имеет доступа к |
|---|---|---|
| `runtime/sensing.py` | целостность пяти видов, tick, энкодер, health | STL, truth, Supervisor |
| `runtime/geometry.py` | слияние точек, OBB, поперечные сечения, K | имя товара, каталог SKU |
| `runtime/uncertainty.py` | интервалы, abstain, безопасный маршрут | evaluator и ожидаемый класс |
| `runtime/scheduling.py` | правила, `item_seq`, immutable commit, encoder deadline | wall-clock таймер маршрута |
| `runtime/mechanics.py` | FSM заслонок, E-stop, выходы, reset | эталон класса |
| `evaluator/oracle.py` | точная геометрическая истина и proxy-аудит | управление приводами |
| `evaluator/supervisor.py` | корреляция внешнего `item_seq`, метрики после действия | канал команд runtime |
| `contracts/` | неизменяемая схема события | бизнес-логика обеих сторон |

## Инварианты безопасности

- B — единственный маршрут, требующий двух положительных механических разрешений.
- Неизвестные габариты → `ABSTAIN_DIMENSION` и C; неизвестная форма → `ABSTAIN_SHAPE` и D.
- Потеря питания возвращает заслонку габаритов к C, заслонку формы к D.
- При потере энкодера нельзя перейти на расчёт по настенным часам.
- Неполный `FrameBundle` нельзя дополнить кадром прошлого тика.
- Решение и физический выход — разные поля события.
- `SUCCESS` без датчика выхода запрещён схемой и FSM.
- Остановка evaluator не открывает B и не изменяет семантический хэш решения.

## Защита от oracle leakage

`config/architecture.json` запрещает runtime использовать `Supervisor`, `Receiver`, пакет evaluator, пути исходных моделей и обратный канал. `tools/check_architecture.py` проверяет импорты, API, устройства и конфигурацию; `tools/leak_test.py --with-canary` доказывает, что намеренная запрещённая ссылка делает проверку красной.

Evaluator получает событие после действия runtime и только тогда добавляет ожидаемый класс. Переименование STL/DEF и остановка evaluator не меняют решение на одинаковом sensor replay.

## Браузер и Webots

`web/` — отдельный статический клиент для удобства жюри. Он локально разбирает STL, показывает геометрию, правило и понятную анимацию маршрута. Он не запускает RangeFinder и не является доказательством физики.

Авторитетный контур — Webots в digest-pinned Docker-образе без runtime-сети. Только там проверяются движение по конвейеру, контакты, две заслонки, пассивный возврат, E-stop, ZPA и датчики выходов.

## Границы системы

В проект не входят реальный ПЛК, промышленная камера, сертификация функциональной безопасности и испытания на физическом конвейере. Модель проверяет алгоритм и инженерную конструкцию в цифровой среде; перенос на оборудование потребует отдельной калибровки, HIL/SIL и промышленной оценки рисков.

# Цифровая симуляция SafeSort

## Что является виртуальным конвейером

Конвейер — это мир Webots `webots/worlds/smoke_unknown_stl.wbt`, а не STEP-файл и не браузерная карточка. Мир содержит площадку, зоны A/B/C/D, ленту, товар, пять датчиков глубины, энкодер, две сортировочные заслонки и датчики выходов. `webots/protos/FailSafeGate.proto` описывает механику заслонки.

STL и STEP относятся к товару:

- STL хранит триангулированную поверхность и подходит для визуализации/симуляции;
- STEP хранит точную редактируемую CAD-геометрию;
- ни один из форматов сам по себе не описывает весь сортировочный узел.

## Авторитетное демо и удобный демо-клиент

### Кандидат на Webots physical smoke

Так должен выглядеть минимальный физический proof: неизвестный/переименованный STL появляется в A, движется по ленте, runtime получает виртуальные сенсорные данные, формирует решение, воздействует на заслонку и получает подтверждение выхода. Удаление подтверждения выхода должно переводить сценарий в `FAULT` и делать команду красной. Исторические трассы до исправления ориентации RangeFinder не считаются финальным доказательством; статус остаётся «не заявлено» до новых B/C/D и missing-exit прогонов с актуальным calibration hash.

Запуск:

```powershell
python run_scenario.py run --scenario scenarios/smoke/unknown_stl_b.yaml --seed 42 --output artifacts/smoke
python run_scenario.py verify --bundle artifacts/smoke
```

Физический контур Webots сейчас ограничен smoke-сценариями; релизный physical proof ещё не заявлен до post-calibration gate. Численные reliability, hour-flow и power-loss harness ни при каких условиях не называются тысячами Webots-маршрутов.

### Браузерный STL-демонстратор

Статическая страница `web/` загружает файл локально, без отправки на сервер, визуализирует mesh и объясняет решение. Она нужна для быстрой проверки собственного файла жюри и для пяти пользовательских потоков: обычный STL, переименованный неизвестный, граничный, некорректный и повтор после ошибки.

Браузер может показать анимацию товара по схеме конвейера, но это объясняющая визуализация. Она не моделирует контакты, RangeFinder, массу, пружину и датчики Webots.

## Геометрия площадки

Scene validator проверяет:

- рабочую область 6000×10000 мм;
- фиксированные координаты A и B;
- зоны C и D размером 1200×800×800 мм внутри площадки;
- ленту шириной 500 мм и высотой 700 мм;
- входной товар в пределах 500×500×500 мм.

Первичным доказательством является `scene-validation.json`, а не скриншот.

## Сенсорная модель

Пять RangeFinder направлены сверху, слева, справа, спереди и сзади. `FrameBundle` допустим, только если:

- все включённые виды присутствуют;
- у всех один simulation tick;
- у всех один encoder tick;
- health каждого вида — `HEALTHY`;
- хэш калибровки совпадает с закреплённым.

Шум, dropout и motion compensation зависят от seed и записываются в manifest. Старый кадр нельзя выдать за текущий. Нижний экспериментальный вид отключён в release, потому что заранее заданный порог полезности не достигнут.

## Геометрический алгоритм

1. Калиброванные точки пяти видов объединяются.
2. Грубый AABB быстро выявляет очевидный C.
3. PCA строит ориентированную систему координат.
4. Около официальных границ выполняется уточнение OBB по углу.
5. На нескольких плоскостях и квантилях ищутся замкнутые сечения.
6. Для лучшего сечения вычисляется официальный `K = min(radius)/max(radius)`.
7. Coverage, closure и максимальный gap могут дать типизированную неопределённость вместо выдуманного числа.

Runtime получает только `FrameBundle` и калибровку. Известные SKU, имя STL и truth в интерфейсе алгоритма отсутствуют.

## Механика сортировки

Две заслонки стоят последовательно:

```text
A / sensing → [dimension gate; покой к C] → [shape gate; покой к D] → B
                              ↘ C                         ↘ D
```

Заслонка содержит RotationalMotor, PositionSensor, пружину, демпфер, жёсткие упоры и ограничение момента. Первая отводит C и требует permit для движения дальше. Вторая отводит D. Для B нужны оба position-confirmed permit.

ZPA (управление без накопления давления) удерживает следующий pocket при fault. E-stop снимает drive не позднее двух simulation steps и требует reset.

## Типы испытаний

| Профиль | Что он действительно проверяет | Тип доказательства |
|---|---|---|
| smoke `unknown_stl_b` | кандидат на полный цикл и exit canary | Webots physical только после post-calibration gate |
| sensing | синхронизацию, health, калибровку, replay | contract/harness + smoke regression |
| geometry | OBB, сечения, K, invalid cases | synthetic geometry harness |
| uncertainty | интервалы, rescan, risk/coverage | numerical harness |
| gates/power loss | FSM и модель возврата | unit + numerical physics-proxy + smoke regression |
| reliability-release | правила, identity, faults, статистику | numerical reliability harness; не 10k Webots runs |
| hour-flow | очередь, дедлайны, throughput | analytical + discrete-event |
| ablations | сравнение архитектур на одинаковых cases/seeds | paired numerical harness |
| release | сборку согласованных файлов и семантический хэш | evidence packaging/determinism |

## Collision proxy и evaluator

Evaluator хранит геометрическую истину отдельно и проверяет proxy по bbox, объёму, footprint, центру масс и инерции. Он сопоставляет только уже существующий внешний `item_seq` и добавляет truth после действия runtime. Эти данные не возвращаются в контур управления.

Proxy-аудит повышает инженерную честность, но не заменяет контактный Webots-тест каждого товара. В итоговой защите следует показывать отдельно только прошедший post-calibration physical smoke, затем численный стресс, не смешивая их в одно число.

## Ограничения

- Нет подтверждения на реальном оборудовании.
- Длительный 5143/ч профиль пока является discrete-event/аналитическим, не часовым Webots-прогоном.
- Численная модель возврата заслонки не равна 200 отдельным Webots fault-runs.
- Трение, контакт и collision proxy требуют дальнейшей HIL/физической валидации перед промышленным внедрением.
- Браузерный результат служит объяснением и локальной проверкой файла, а не официальным oracle.
